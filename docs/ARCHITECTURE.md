# Architecture & Implementation

This is the full implementation guide for the VoIP + Opus app — what's in the
repo, how it fits together, and why each piece is the way it is. It's intended
to be read top-to-bottom by a new contributor.

Companion documents:
- [concepts/](./concepts/README.md) — four standalone explainers for the
  technologies this doc takes for granted: WebRTC, ICE/STUN/TURN, mesh vs
  SFU, and JWT auth. Read these if you want the *why*; this doc is the
  *what*.
- [PROTOCOL.md](./PROTOCOL.md) — exact wire-protocol message reference.
- [DEPLOY.md](./DEPLOY.md) — running it locally and deploying it for real.

## 1. System overview

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│                            BROWSER  (web/)                                   │
│                                                                              │
│      React 19 + TypeScript + Vite 5 + Tailwind + Framer Motion               │
│      Runs as a PWA — installable, service worker, offline shell              │
│                                                                              │
└──────────────┬──────────────────────────────────────────┬───────────────────┘
               │                                          │
               │ HTTPS                                    │ WSS
               │   POST /api/auth/{signup,login}          │   /ws?token=<JWT>
               │   GET  /api/auth/me                      │   one persistent
               │   GET  /api/users/online                 │   connection
               │   (JWT in Authorization: Bearer …)       │
               ▼                                          ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                              │
│                            SERVER  (server/)                                 │
│                                                                              │
│      FastAPI + uvicorn (async)    +    SQLite (users only)                   │
│      PyJWT (HS256, 7-day)         +    bcrypt (password hashing)             │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘


    Plus a third channel that the server is NOT in:

  ┌──────────────┐                                          ┌──────────────┐
  │  Browser A   │   ◄─── direct UDP, encrypted (SRTP) ───► │  Browser B   │
  │              │       Opus packets, ~50/sec               │              │
  └──────────────┘                                          └──────────────┘
                       ↑
                       Negotiated once via the server during call setup
                       (SDP + ICE candidates). Audio itself NEVER touches
                       the server.
```

Three planes:

| Plane | Carries | Where it lives |
|---|---|---|
| **Control (HTTP)** | Login, signup, `/me`, online roster | TCP/HTTPS, browser ↔ server |
| **Signaling (WebSocket)** | SDP offers/answers, ICE candidates, room membership events | WS over the same HTTPS port, browser ↔ server |
| **Media (RTP/SRTP)** | Real-time Opus-encoded audio | UDP, browser ↔ browser **direct** |

The server never sees a byte of audio. It is a *matchmaker*, not a relay.
Once peers have exchanged SDP + ICE through it, they ship audio straight to
each other; you could turn the server off mid-call and conversations would
continue. The only exception is if both peers are behind hostile NATs and ICE
falls back to a TURN relay — that's a different server (not the signaling one)
and even then it sees only encrypted packets it can't decrypt.

### Key client modules in detail

#### `signaling.ts` — `SignalingClient`

Single WebSocket connection. Auto-reconnects with exponential backoff on
unexpected close. Has an outbox queue so callers can issue methods before the
socket has fully opened; messages are flushed on `onopen`. Speaks every
message type listed in [PROTOCOL.md](./PROTOCOL.md).

Handlers are typed:
```ts
new SignalingClient(token, {
  onRoomJoined: (d: RoomJoinedData) => { ... },
  onParticipantJoined: (d: ParticipantJoinedData) => { ... },
  onOffer: (sdp, fromPeerId) => { ... },
  ...
});
```

#### `webrtc.ts` — `Call`

The 1:1 path. One `RTCPeerConnection` per call. Acquires the mic via
`getUserMedia` (audio only: echo cancellation, noise suppression, AGC),
adds tracks, generates the offer / handles the answer. Exposes `setMuted`,
`getStats`, and `end`. ICE servers are STUN by default + optional TURN from
env vars.

#### `room.ts` — `RoomManager` (the new mesh manager)

The multi-party path. Wraps a `SignalingClient` and maintains a
`Map<peerId, PeerLink>` where each `PeerLink` is one `RTCPeerConnection`,
its remote `MediaStream`, and its current `connectionState`.

Three notable design choices:

1. **One mic stream, fanned out.** `getUserMedia` is called once on join.
   The resulting `MediaStream` is added to every `RTCPeerConnection`.
   Mute toggles the `enabled` flag on the local audio track; every peer
   sees the same change.

2. **Glare avoidance via deterministic initiation.** When two peers see
   each other for the first time, both could try to send an offer ("glare").
   We avoid that with a stable rule: the peer whose `userId` compares LARGER
   is the one that calls `createOffer`. Both browsers agree because both
   know both IDs.

3. **Lazy peer creation.** If an `offer` arrives from a peer we haven't yet
   registered (rare ordering edge case — the offer arrives before the
   `participant-joined` event), the manager lazy-creates a peer link with
   placeholder user info and processes the offer immediately. The user info
   is filled in when `participant-joined` arrives.

#### `opusEncoder.ts` — for `/demo` only, NOT for calls

Wraps the real `opus-recorder` library (which is libopus 1.3.1 compiled to
WebAssembly). The `/demo` page uses it to demonstrate encoding interactively
and to round-trip the audio back through the browser's native Opus decoder.

**The `/call` and `/room` flows DO NOT use this module.** They rely on the
browser's native Opus-over-RTP support built into WebRTC. There's no
JavaScript on the audio hot path during a call.

### Backend modules in detail

#### `config.py`

Frozen dataclass loaded once at startup from environment variables. Defaults
are local-friendly so `uvicorn server.main:app` runs out of the box:

```python
PORT=8000
JWT_SECRET=dev-insecure-change-me
JWT_ALGORITHM=HS256
JWT_EXPIRES_DAYS=7
DB_PATH=./data/voip.db
CORS_ORIGINS=http://localhost:5174,http://127.0.0.1:5174
```

#### `db.py`

Stdlib `sqlite3` with one table. Each call opens its own short-lived
connection — SQLite serializes writes internally, which is fine at this
scale. No ORM, no migrations; the schema is in `SCHEMA` at the top of the
file and applied via `CREATE TABLE IF NOT EXISTS` at startup.

```sql
CREATE TABLE users (
    id            TEXT PRIMARY KEY,
    username      TEXT NOT NULL UNIQUE,
    email         TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    created_at    TEXT NOT NULL
);
```

#### `auth.py`

- `hash_password(plain) → str` — bcrypt with default cost factor.
- `verify_password(plain, hashed) → bool` — constant-time bcrypt compare.
- `create_access_token(user_id) → str` — HS256 JWT, `sub=user_id`, `exp=now+7d`.
- `decode_token(token) → Optional[user_id]`.
- `get_current_user` — FastAPI `Depends` that pulls the bearer token,
  decodes it, looks up the user, and returns the row. 401s on any failure.

#### `ws/signaling.py` — the brain

`ConnectionManager` holds three in-memory maps guarded by one `asyncio.Lock`:

```
_sockets:   user_id   → WebSocket          # currently connected users
_rooms:     room_code → set[user_id]       # who's in each room
_user_room: user_id   → room_code          # reverse for cleanup
```

The `/ws` endpoint:

1. Calls `_authenticate(ws)` — parses `?token=<JWT>`, decodes, looks up
   the user. Closes with code 1008 on failure.
2. `manager.register(user_id, ws)` — replaces any prior connection from
   the same user (handles reconnects gracefully).
3. Sends `websocket-connected` + broadcasts the new roster.
4. Enters the read loop, parses each message as JSON, dispatches via
   `_route(user, msg)`.
5. On disconnect: implicit `room-leave` + broadcast `participant-left` to
   remaining members + roster update.

Every signaling message is routed through `_route`. For peer-addressed
messages (`offer`, `answer`, `ice-candidate`, `hang-up`, `call-response`),
the server enforces that the sender and target are in the same room (or
that the sender is initiating a 1:1 call). Cross-room signaling is silently
dropped.

## 5. Data model

Persistent storage is one SQLite file, one table:

```sql
CREATE TABLE users (
    id            TEXT PRIMARY KEY,    -- uuid4
    username      TEXT NOT NULL UNIQUE,
    email         TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,       -- bcrypt
    created_at    TEXT NOT NULL        -- ISO 8601 UTC
);
CREATE INDEX idx_users_username ON users(username);
CREATE INDEX idx_users_email    ON users(email);
```

Everything else lives in memory in `ConnectionManager`:

| State | Shape | Lifetime |
|---|---|---|
| Online users | `dict[user_id, WebSocket]` | One WS connection |
| Rooms | `dict[room_code, set[user_id]]` | Until last member leaves |
| Per-user room membership | `dict[user_id, room_code]` | Same |

This is deliberate. Rooms are ephemeral — they exist for the duration of a
call. There's no "list past rooms" or "rejoin previous room" feature, so
persisting them would add complexity for no user-visible value.

When/if we need persistence later, candidate additions are:
- `rooms(id, code, created_at, owner_id)` — for naming + permissions
- `call_log(room_id, user_id, joined_at, left_at)` — for history
- `contacts(user_id, contact_id, added_at)` — friends list

## 6. Trust boundary

### HTTP API
- Every protected route requires a Bearer JWT in `Authorization`.
- Token is HS256 JWT, signed with `JWT_SECRET`, 7-day expiry, `sub = user_id`.
- Verified on every request via the `get_current_user` FastAPI dependency.

### WebSocket
- JWT supplied as `?token=` query param (browsers can't set headers on WS
  upgrades).
- Validated **once** at handshake; the WS is closed with code 1008 if the
  token is missing, malformed, or its user is gone.
- After handshake the connection is trusted as the authenticated user; the
  server tags every routed message with `from: <authenticated user id>`,
  preventing a client from spoofing identity.

### Passwords
- bcrypt, library default cost factor (currently 12).
- Never stored, logged, or sent in plaintext after signup/login.

### Media
- All RTP between peers is encrypted with SRTP using a key negotiated in the
  DTLS handshake during call setup. The server cannot decrypt any audio
  packet even if it could intercept one.

### CORS
- Configurable allow-list via `CORS_ORIGINS` env. Default for local dev is
  `http://localhost:5174,http://127.0.0.1:5174`. In production, restrict to
  your actual frontend origin.

### Surface area
- 4 HTTP routes, 1 WebSocket endpoint, 1 health endpoint. That's it.
  Less surface = fewer ways to mess up.

## 7. Why these choices

**FastAPI** over stdlib HTTP server: async-native, no threading hazards,
native WebSocket support, Pydantic validation for free, clean auth dependency
injection. The old codebase mixed `http.server` (threaded) with `websockets`
(async); that race condition was the source of half the original bugs.

**SQLite** over Postgres for v1: the only persistent data is the users table.
SQLite is one less moving part; swap for Postgres when the data model grows.
The `db.py` module is small enough that the swap is a one-day job.

**No ORM**: with one table, SQLAlchemy / SQLModel are pure overhead. The
queries are 8 lines total.

**One React app** instead of two: the old repo had a half-built call client
and a separate educational tutorial. One Vite app + one router + one design
system removes the worst category of bugs (drift between two near-identical
codebases).

**Mesh topology** over SFU for v1: up to ~5 participants, mesh is the right
choice. Zero server-side audio infrastructure. Every browser does its own
encoding. No additional service to deploy or scale. Beyond ~5 participants
mesh starts hurting CPU and uplink bandwidth (each peer encodes once but
*sends* N-1 streams); at that point an SFU like mediasoup or LiveKit becomes
worth the extra service.

**WebRTC's built-in Opus** for calls: the browser already has a tightly
integrated, hardware-accelerated, jitter-buffered Opus codec. Encoding in
JavaScript would be slower and lower quality with no upside. The
`opus-recorder` WASM module exists purely for `/demo`, where the goal is
educational visibility, not call quality.

**JWT in `localStorage`** for the PWA: enables persistence across PWA launches
and tab restarts. Trade-off: vulnerable to XSS-driven theft. Mitigated by
strict CSP at deploy time and the fact that no third-party scripts run in
this app. If/when third-party scripts are added, migrate to httpOnly cookies.

### Manual smoke test

1. Open <http://localhost:5174>; verify Landing renders.
2. Walk through `/learn`; record yourself in Section 2 ("turn your voice
   into numbers"); verify the sample-dot zoom view shows individual PCM
   samples.
3. On `/demo`: record, set bitrate to 8 kbps and 128 kbps; verify the file
   sizes differ ~20×, the playback sounds appropriately worse at low
   bitrate, and the "Show code" panel reflects the current settings.
4. Sign up two users in two browsers (or browser + phone on the same LAN).

   invite link, paste in the second browser.
6. Verify both browsers show each other in the participant list, the
   connection-state pill turns "connected", and audio flows both ways.
7. Add a third browser to the same room; verify all three see each other
   and the mesh works.
8. Close one tab; verify the others see `participant-left` and the call
   continues for the remaining two.

## 10. What this design does NOT do (yet)

- **No persistent contacts / friends list.** The online roster is in-memory
  and broadcast on every connect/disconnect.
- **No call history.** Rooms are ephemeral; nothing about them is logged
  past the moment the last member leaves.
- **Group rooms cap out around 5 participants** before mesh CPU/bandwidth
  pressure becomes annoying. To go higher you'd add an SFU (mediasoup,
  LiveKit, Janus, or roll your own with `Pion`).
- **No video.** Audio-only by design. Adding video means changing
  `getUserMedia` constraints and adding a `<video>` tag per peer in the
  Room UI — about a day's work, but bandwidth and battery costs are
  substantial.
- **No push notifications.** A user with the PWA closed cannot be notified
  of an incoming call. Real fix is the Web Push API + a service-worker
  notification handler + storing push subscriptions in the DB; about a
  week of work including iOS-specific quirks.
- **No LAN-offline bundle.** WebRTC works on a LAN without internet, but
  packaging the whole thing as a single binary with mDNS + a bundled local
  CA is its own project. Discussed in earlier design notes; not built.
- **No TURN server configured by default.** STUN is hardcoded
  (`stun:stun.l.google.com:19302`). TURN config is read from env when
  present but no relay is provisioned. For most home/office networks this
  is fine; for symmetric NATs or corporate firewalls, calls will fail
  silently. Add a TURN provider in production.
- **No load testing.** The signaling server is async and should easily
  handle hundreds of concurrent users on one box, but we haven't actually
  measured it.
